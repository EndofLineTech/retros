## Section 3 — What the Agent Got Wrong

**Inheriting the `main` target for PR #90 without questioning it.** When PO accepted architect's Option (b) — split the bloated PR #82 into a standalone ADR-doc PR — I briefed the PE agent to cherry-pick onto a branch *from `main`* and target `main`. That was pure inheritance from PR #82's original target, not a reasoned choice. The ADR-001 acceptance was already on `dev` at commit `1b9e2623`; no main-branch consumer required immediate promotion; and the whole point of architect's Option (b) was to *avoid* release-cut-discipline antipatterns — which is exactly what cherry-picking a single doc file to main recreated at smaller scale. PO caught it with a one-line question: *"Why does 90 need to go to main?"* Cost: one agent run, one closed PR, main branch protection blocked the merge, and PO spent attention on a question I should have answered before briefing. The more general failure mode here — inheriting a decision from a predecessor artifact without re-deriving it under the new scope — is one I want to be watchful for; saved to memory as `feedback_question_pr_target.md`.

A secondary one: my CodeQL alert triage was incomplete. I queried `per_page=30` and reported "37 open alerts" to PO. The Security Engineer agent's deeper read surfaced 14+ additional HIGH-severity findings I missed (8 × stack-trace-exposure, 1 × reflective-xss, 2 × partial-ssrf, plus the export.py path-injection cluster). If PO had made a merge decision based on my first summary instead of holding for SE triage, that would have been a real hole.

**Decision prompts were overloaded with prior context.** PO fed this back directly: *"we get overly complex sometimes; when we get to decision points, there's a ton of prior context I have to go back and read through; I need better summarization of the issues."* Concrete examples from this session: (1) the PR #82 merge-strategy decision came with a two-row options table, architect comment excerpts, SE comment excerpts, four bead IDs to cross-reference, and a recommendation — PO had to go back multiple messages to reconstruct what architect vs. SE were disagreeing about. (2) The three-decision batch (merge strategy / ADR numbering / trace_id fix) was dense enough that each option referenced state from 3+ messages earlier. (3) Sprint-status updates listed every bead and every PR state in a table — comprehensive, but not summarized for the decision-maker. The fix is on me: at decision time, compress prior context to a 1-2 line recap ("here's the state + here's what's new since last turn"), then state the decision at hand, then present 2-3 options with one-line trade-offs. The comprehensive summary belongs in a retro, not in the middle of a decision prompt. Saving this as a durable lesson.

**Retro chat summary buried the PO-facing feedback.** When I delivered the retro summary in this session's chat, I led with agent failures (Section 3) and didn't surface Section 2 at all. PO's response: "There's nothing about how I could do better." Section 2 existed in the file but was invisible in the chat. ## Section 4 — What Would Make the Project Better

**A way to update in-flight background agents with corrected context.** During the session, the QA flake-audit (tp681) agent concurrently discovered that the trace_id failures were *order-dependent flakes*, not deterministic failures as I'd told the DBA agent who was actively trying to fix PR #87. I wanted to send a correction — `ToolSearch` returned nothing for `SendMessage`. The DBA reconverged on the correct diagnosis independently (and ultimately found a production observability bug), but the correction would have saved cycles. More generally: when multiple parallel agents discover things that invalidate sibling agents' premises, there's no graceful broadcast. Two ways to handle it: (1) add a real async messaging/update channel for background agents, or (2) PM discipline of "brief all agents to self-verify their premises before trusting my framing." Absent the tool, the discipline is the workaround.

A secondary improvement: **the code reviewer persona was never explicitly invoked on any of the 8 merged PRs**. Author-merges-own-PR is the de-facto shipping pattern, which is fine most of the time but less fine for things like PR #87's production observability fix. Putting a code-reviewer gate on specific PR classes (security fixes, production regressions, migrations) would catch a class of issues the current flow relies on author self-discipline for.

## Section 6 — Lessons

- **Keep**: Numbered-option decision prompts to PO ("1 / 2 / 3 — which?"). Respected PO time and elicited crisp answers. PO answered with just the option number multiple times; the format works.
- **Keep**: Worktree-isolated + background + persona-scoped parallel agent dispatches. Seven agents in flight simultaneously with no filesystem collisions beyond one benign rebase issue (docs/runbooks/ between bwly4 and dl1bd, resolved at merge).
- **Keep**: Honoring the "close bead only after merge" discipline. Every agent deferred `bd close` to PM judgment after PR merged. No premature closures, no orphan beads.
- **Stop**: Inheriting target branches / prior artifact decisions without re-deriving them under the new scope (captured as memory `feedback_question_pr_target.md`).
- **Stop**: PM executing work that belongs to execution personas. Role drift is subtle; the line is "board updates = PM, file/code edits = execution persona" (saving as memory).
- **Stop**: Reporting findings from paginated APIs without checking pagination. CodeQL missed 14+ HIGH findings on my first query.
- **Start**: Briefing agents to self-verify the premises in my brief. When a sibling agent discovers something that invalidates another agent's brief, there's no in-flight correction channel. Agents should default to verifying their own premises.
- **Start**: Explicit release-cadence framing at sprint planning. Ask PO directly: "is a release cut part of sprint exit? If so, at what version?"
- **Start**: Request code review on specific PR classes — production regressions, security fixes, schema migrations — even when the author is a capable agent. Single-author-merges-own-PR is fine for scaffolds, weaker for high-stakes changes.
